# Aion Forge 版本重排方案（从 v0.2.1 起）

## 背景

当前仓库存在“本地 tag 前进到 `v0.6.0`，但远端公开版本仅到 `v0.2.1`”的分叉。  
为避免对外版本认知混乱，采用“对外口径重排、内部 tag 保留”的策略。

## 事实基线（已确认）

- 远端已有版本：`v0.0.1`、`v0.1.0`、`v0.2.0`、`v0.2.1`
- 本地未发布 tag：`v0.4.5-pre-router`、`v0.5.0-route-task`、`v0.5.1-final`、`v0.5.2-tri-engine`、`v0.5.2`、`v0.6.0`

## 重排目标

1. 对外版本号连续、可解释  
2. 版本号与用户可见能力匹配  
3. 保留历史追溯能力（不丢失 commit 信息）

## 推荐口径（对外）

### 方案 A（推荐，低风险）

- 继续以 `v0.2.1` 为“最后公开版本”
- 将后续真实能力集合重新发布为：
  - `v0.2.2`：稳定性修复与编排修正（原 `v0.5.1`/`v0.5.2` 的修复向内容）
  - `v0.3.0`：`route_task` + tri-engine + health_check + all-engines-failed 检测（用户可见能力包）

说明：

- 不对外引用 `v0.4.5`~`v0.6.0`，这些仅作为内部历史标签。
- 在 release note 明确“版本口径重排”，避免用户困惑。

### 方案 B（保留现有编号，较高认知成本）

- 直接推送 `v0.4.5`~`v0.6.0` 并逐个补发布说明  
- 需要解释为何跳过 `v0.3.x` 且历史未公开，外部理解成本更高

## 执行动作清单（按顺序）

1. 冻结当前发布口径：确定采用方案 A 或 B  
2. 写统一发布说明：说明“历史 tag 与公开版本的差异”  
3. 按 `docs/RELEASE_CRITERIA.md` 补齐指标与示例  
4. 打新公开标签（推荐 `v0.2.2` 或 `v0.3.0`）  
5. 推送标签并创建 GitHub Release  
6. 在 README/CHANGELOG 顶部写明“当前公开稳定版本”

## 建议的当前落地

基于现状，建议直接执行：

- 当前分支合入后，先发 `v0.3.0`
- 发布内容包含：
  - route_task 路由能力
  - tri-engine 协作编排
  - health_check
  - text_wordcount
  - all-engines-failed 检测
- 同步给出 2 项指标：成功率、P95 延迟（对比 `v0.2.1`）

## 风险与规避

- 风险：团队成员继续沿用旧本地 tag 口径  
  - 规避：在 PR 模板/发版流程里强制引用公开版本号
- 风险：用户看到提交信息中的 `v0.6.0` 描述产生疑惑  
  - 规避：release note 中明确“历史内部命名，不代表公开版本序列”
