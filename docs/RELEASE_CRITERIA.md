# Aion Forge 发版门槛（Release Criteria）

## 目的

避免“版本号增长快，但用户感知增强不足”。  
从本规范起，版本发布以**用户可见能力**和**可量化质量**为准，不以重构工作量为准。

## 版本语义（执行版）

- `MAJOR`（X.0.0）：架构或协议不兼容变更，或迁移成本高。
- `MINOR`（0.X.0）：新增面向用户的能力（可演示、可验收）。
- `PATCH`（0.0.X）：修复与稳定性提升，不新增主能力。

当前项目处于 `0.x` 阶段，但依然执行上述区分，禁止“把 patch 级改动发成 minor”。

## 发布硬门槛（必须全部满足）

### 1) 能力门槛（用户视角）

每个版本必须至少满足以下之一：

- 新增 `>= 1` 个可直接调用的能力（CLI / MCP / HTTP 任一入口可见）；或
- 对现有能力带来可验证的行为增强（输入输出或成功路径明显变化）。

并且必须提供：

- 示例调用（至少 1 条）；  
- 预期输出示例（至少 1 个）。

### 2) 质量门槛（可量化）

每个版本必须给出至少 2 个指标（至少包含 1 个稳定性指标）：

- 成功率（success rate）
- P95 延迟
- 超时率 / 失败率
- 资源占用（可选）

要求：

- 指标有基线对比（相对上个已发布版本）
- 指标来源可追溯（测试日志或压测记录）

### 3) 变更边界门槛（版本匹配）

- 仅修复/重构/文档：只能发 `PATCH`
- 新增用户能力：可发 `MINOR`
- 存在 breaking change：必须标注并走 `MAJOR` 或明确迁移说明

## 发布前检查清单（Release Checklist）

发布人必须逐项确认：

1. 已定义版本类型（major/minor/patch）且符合变更边界  
2. 已列出用户可见能力变化（不是内部实现细节）  
3. 已给出至少 1 条示例调用与输出  
4. 已提供至少 2 个量化指标（含稳定性项）  
5. `CHANGELOG.md` 已更新并按“新增/修复/破坏性变更”分类  
6. 关键测试已通过（最少包含改动相关 crate 测试）  
7. 已确认密钥与本地配置未入库（如 `.mcp.json`）  

## 禁止项

- 禁止将“仅重构/清理”包装成 `MINOR`
- 禁止仅凭提交量/文件数提高版本号
- 禁止在无指标对比时宣称“性能提升/稳定性提升”

## 推荐发版节奏

- 功能上线：按能力包发 `MINOR`
- 快速修复：按缺陷批次发 `PATCH`
- 复杂规划中的中间节点：只保留在分支，不打 release tag
