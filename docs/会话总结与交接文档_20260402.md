## 二、当前已知问题（hermes 审计发现）

### B1: rag_query AI 不可用时返回空回答（严重度：高）
- **现象**：rag_ingest 入库成功（27 chunks），rag_query 检索到 3 个 chunk，但回答为"无法生成回答"
- **原因**：AI 后端（智谱 glm-4.7-flash）限流时，rag_query 直接返回空而非原始 chunks
- **修复方案**：rag_query 在 AI 不可用时应返回原始 chunks 内容，让调用方自己判断
- **文件**：`aion-router/src/builtins/rag.rs`

### B2: route_task 关键词覆盖不足（严重度：中）
- **现象**：常见任务描述（"分析日志""写个脚本""测试代码"）全进 passthrough
- **原因**：17 条规则的关键词偏专业术语，缺少口语化表述
- **修复方案**：补充关键词——日志/log/排错/debug → code-daily；脚本/script/自动化 → code-daily；测试/test → code-review
- **文件**：`router.json`

### B3: 安全过滤器误拦正常文本（严重度：中）
- **现象**：regex_match 处理含"token expired"的日志文本时被安全审查拦截
- **原因**：安全层检查参数值中的敏感词（token/key/password），不区分是凭证还是普通文本
- **修复方案**：安全审查应只检查参数名和顶层字段，不深入检查参数值中的子字符串
- **文件**：需定位安全审查逻辑所在位置（可能在 aion-router/src/executor.rs 或 orchestrator.rs 的 security review 部分）

## 三、架构决策记录

### 三引擎真实调用——拆为独立技能包
- **原因**：MCP 协议 60 秒超时写死在 Claude Code 里，三家 CLI 冷启动+推理需要 30-90 秒
- **现状**：代码保留在 orchestrator.rs，但 .mcp.json 设 AI_PASSTHROUGH=true，不走真实调用
- **后续**：做独立技能包时，通过 PowerShell 直连（非 MCP）调用，不受 60 秒限制

### passthrough 模式——当前默认
- AI 编排工具（ai_parallel_solve 等）返回结构化指令，宿主 LLM 自行处理
- 稳定，无超时风险
- 缺点：不是"真三引擎"，实际上是单引擎

### 部署纪律
- 只有 CC（Claude Code）有权编译部署
- GPT 改源码后交给 CC 编译
- 每次部署必须更新 VERSION.json（SHA256）
- 源码只在 `aion_forge_handoff_v2.1`，不是 `skill_router`

## 六、从《实践论》《矛盾论》得到的方法论启示

### 主要矛盾
用户需要的是"能帮我开发小软件的助手"。aion-forge 的主要矛盾不是工具数量不够，而是**工具能被调用但不能完成真实任务**。

### 实践检验
任何功能上线前，必须用它完成一个真实任务（不是"echo 返回了"就算通过）。建议验收标准：用 aion-forge 工具链端到端完成一个小需求。

### 闭环思维
当前是开环系统——route_task 选工具 → 执行 → 结束。缺少反馈环：执行后验证结果是否正确、不正确则换工具重来。这是下一个版本应该做的核心能力。

### 不要速胜
先让一个场景彻底跑通（比如：解析配置+生成代码+验证结果），再扩展到更多场景。不要追求 49 个工具全部完美。
